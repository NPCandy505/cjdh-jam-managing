# Jam Day 관리 사이트 마스터 플랜

## 1. 프로젝트 한 줄 설명

재즈 동아리 구성원이 **쉽게 곡을 신청하고**, 운영진이 **한눈에 신청 현황을 확인**할 수 있는 잼 데이 관리 웹사이트를 만든다.

이 문서는 첫 구현에 필요한 목표와 범위를 정리한 작업 문서이다.  
현재 기준으로 1차 기획 확정이 반영되어 있으며, 이후 구현 과정에서 세부 문구와 UI는 조정할 수 있다.

---

## 2. 프로젝트 목표

### 핵심 목표

1. 동아리원이 곡 신청을 빠르게 할 수 있어야 한다.
2. 신청된 곡 목록이 표 형태로 명확하게 보여야 한다.
3. 신청자 본인의 역할과, 해당 곡에 나중에 합류한 멤버 정보를 구분해서 보기 쉽게 정리할 수 있어야 한다.
4. 원클릭 버튼을 통해, 시작이 어렵더라도 접근할 수 있는 대안 경로가 있어야 한다.

### 비핵심 목표

1. 처음부터 복잡한 권한 관리 시스템을 만들지 않는다.
2. 처음부터 카카오톡 연동까지 완성하지 않아도 된다.
3. 처음부터 모바일 앱을 만들지 않는다.

---

## 3. 대상 사용자

### 주요 사용자

1. 잼 데이에 곡을 신청하는 동아리원
2. 신청 현황을 관리하는 운영진

### 사용자 특성 가정

1. 대부분은 모바일 메신저와 웹 사용에는 익숙하다.
2. 사이트 이용 시간은 짧고, 빠르게 신청만 하고 나가고 싶어할 가능성이 크다.
3. 긴 설명보다 직관적인 버튼과 폼이 더 중요하다.

---

## 4. 해결하려는 문제

현재 잼 데이 곡 신청이 메신저, 문서, 구두 전달 등으로 흩어져 있다면 다음과 같은 문제가 생길 수 있다.

1. 누가 어떤 곡을 신청했는지 한 번에 보기 어렵다.
2. 신청자가 어떤 세션으로 참여하려는지, 다른 멤버가 누가 합류했는지 파악하기 어렵다.
3. 아직 곡을 못 정한 사람은 신청 진입 장벽을 느낄 수 있다.
4. 운영진이 신청 현황을 정리하는 데 시간이 든다.

이 사이트는 위 문제를 줄이기 위한 **간단하고 실용적인 신청 보드**를 목표로 한다.

---

## 5. MVP 범위

MVP는 "실제로 동아리에서 한 번 써볼 수 있는 최소 기능"에 집중한다.

### MVP에 포함할 기능

1. 곡 신청 현황 표 조회
2. 새 신청 행 추가
3. 필수 입력값 검증
4. 추가 참여 멤버 정보가 비어 있으면 표에서 숨김 처리
5. easy 곡 추천 리스트 표시
6. 원클릭 신청 버튼 배치

### MVP에서 제외할 기능

1. 로그인 / 회원가입
2. 관리자 권한 시스템
3. 실시간 알림
4. 카카오톡 실제 연동
5. 신청 수정 이력 관리
6. 복잡한 검색 / 필터 / 정렬

## 6. 핵심 화면 구성

### A. 상단 소개 영역

목적:
사이트의 용도와 사용 방법을 짧게 안내한다.

포함 요소:

1. 사이트 제목
2. 짧은 설명 문구
3. "아래 표를 확인하고, 곡이 없으면 바로 신청하세요" 같은 안내

### B. 곡 신청 현황 표

목적:
현재 신청된 곡과 참여 현황을 한눈에 보여준다.

표 컬럼:

1. 곡명
2. 신청자
3. 희망 세션
4. 추가 참여 멤버
5. 비고(유튜브 링크 등)

표시 규칙:

1. `곡명`, `신청자`, `희망 세션`은 필수
2. `추가 참여 멤버`가 비어 있으면 셀 내용을 표시하지 않거나, 아예 빈 값으로 둔다
3. `비고`에는 유튜브 링크를 넣을 수 있다
4. 링크가 있으면 클릭 가능한 형태로 보여준다
5. `추가 참여 멤버`에는 신청자를 제외한 참여자의 이름과 세션이 함께 들어간다
6. 예: `신동주(Drum), 박시은(Vocal)`
7. `희망 세션`은 아래 고정 목록 중 하나를 선택한다: `보컬`, `키보드`, `기타`, `베이스`, `드럼`, `색소폰`, `기타 악기`

추가 고려:

1. 모바일에서는 표가 너무 넓어질 수 있으므로 가로 스크롤 또는 카드형 보조 표현이 필요할 수 있다

### C. 신청 추가 영역

목적:
사용자가 자유롭게 행을 추가하도록 돕는다.

핵심 방향:

1. 사용자는 "표를 직접 편집"하기보다 "입력 폼 작성 후 추가" 방식이 더 직관적일 가능성이 높다
2. 입문 개발 관점에서도 폼 제출 방식이 구현과 검증이 쉽다

입력 항목:

1. 곡명 `필수`
2. 신청자 `필수`
3. 희망 세션 `필수`
4. 추가 참여 멤버 `선택`
5. 비고(유튜브 링크 등) `선택`

버튼:

1. `신청 추가`

입력 UX 아이디어:

1. `희망 세션`은 자유 입력이 아니라 고정 드롭다운 기반으로 제공한다
2. `추가 참여 멤버`는 이름과 세션을 한 쌍으로 관리해야 한다
3. 희망 세션 목록: `보컬`, `키보드`, `기타`, `베이스`, `드럼`, `색소폰`, `기타 악기`
4. 초기 MVP에서는 신청 시 추가 참여 멤버를 비워 둘 수 있다
5. 추후에는 다른 회원이 표를 보고 해당 곡에 자신을 추가할 수 있게 확장할 수 있다

권장 표현:

1. `희망 세션`: 신청자가 맡고 싶은 세션 1개 선택
2. `추가 참여 멤버`: `이름 + 세션` 묶음 목록

### D. easy 곡 리스트 영역

목적:
곡을 못 정한 사용자에게 바로 참고할 수 있는 추천 곡 목록을 제공한다.

포함 요소 예시:

1. easy 곡 제목
2. 간단 메모
3. 유튜브 링크 또는 참고 링크

초기 운영 방식:

1. 운영진이 직접 고정 리스트를 관리한다

추후 확장:

1. 난이도 태그
2. 분위기 태그
3. 편성 추천

### E. 원클릭 신청 영역

목적:
그래도 곡을 못 고르는 사람에게 최소 마찰 경로를 제공한다.

예상 기능:

1. 버튼 클릭 시 "운영진에게 추천곡 신청 요청"이 접수된다
2. 초반에는 임시로 사이트 내부 메시지 또는 더미 완료 화면만 제공한다
3. 이후 카카오톡 알림으로 확장한다

예시 버튼 문구:

1. `곡 추천받고 신청하기`
2. `운영진에게 추천곡 요청`
3. `원클릭 잼 신청`

## 8. 데이터 구조 초안

초기에는 단순한 구조로 시작한다.

```ts
type JamEntry = {
  id: string;
  songTitle: string;
  applicantName: string;
  desiredSession: string;
  additionalParticipants?: {
    name: string;
    session: string;
  }[];
  noteUrl?: string;
  createdAt: string;
};

type EasySong = {
  id: string;
  title: string;
  memo?: string;
  youtubeUrl?: string;
};

type OneClickRequest = {
  id: string;
  requesterName: string;
  preferredSession: string;
  memo?: string;
  createdAt: string;
};
```

### 해석

1. `desiredSession`은 신청자가 맡고 싶은 세션
2. `additionalParticipants`는 신청자를 제외한 추가 참여 멤버 목록
3. 각 추가 참여 멤버는 `이름 + 세션`으로 저장한다
4. `noteUrl`은 주로 유튜브 링크 용도
5. `OneClickRequest`는 MVP 기준 `참여자`, `세션`, `비고`를 받는다

## 9. 추천 기술 스택

웹 개발 입문자 기준으로 **너무 무겁지 않고, 자료가 많고, 범용적인 조합**을 우선한다.

### 권장 스택

1. `Node.js`: 개발 서버를 실행하고, 프로젝트 전체가 동작하는 자바스크립트 런타임
2. `Next.js`: 화면, 서버 기능, 폼 처리, API를 한 프로젝트 안에서 만들 수 있게 해 주는 웹 프레임워크
3. `TypeScript`: 자바스크립트에 타입을 더해 실수를 줄이고, 협업과 유지보수를 쉽게 해 주는 언어 도구
4. `SQLite`: 곡 신청 정보, easy 곡 목록, 원클릭 신청 내역을 저장하는 가벼운 데이터베이스
5. `Prisma`: 코드에서 데이터베이스를 더 쉽게 읽고 쓰도록 도와주는 ORM
6. `Vercel`: Next.js 프로젝트를 가장 간단하게 배포하고 운영하기 좋은 호스팅 서비스

### 이 스택을 권장하는 이유

1. JavaScript/TypeScript 기반이라 자료가 많다
2. 프론트엔드와 간단한 서버 기능을 한 프로젝트에서 처리할 수 있다
3. Next.js는 폼 제출, API 라우트, 배포가 비교적 단순하다
4. 소규모 프로젝트에서 시작하기 좋다

### 현재 추천 결론

1차 목표가 "빠르게 만들어서 실제로 써보기"라면  
**`Node.js + Next.js + TypeScript + SQLite + Prisma + Vercel`** 조합을 사용한다.

이 조합에서의 역할 분담은 다음과 같다.

1. `Node.js`: 로컬 개발 환경의 실행 기반
2. `Next.js`: 사이트 화면과 서버 로직 담당
3. `TypeScript`: 코드 안정성 담당
4. `SQLite`: 데이터 저장 담당
5. `Prisma`: 데이터베이스 연결 및 조회/저장 담당
6. `Vercel`: 배포 및 운영 담당

## 10. 기능 요구사항

### 필수 기능

1. 사용자는 곡 신청 정보를 입력할 수 있다
2. 신청 후 목록에서 바로 확인할 수 있다
3. 필수 항목이 비어 있으면 제출되지 않는다
4. 비고 링크가 있으면 클릭 가능해야 한다
5. 제출된 신청곡에 "추가 참여 멤버" 열에 대한 값을 추가할 수 있어야 한다
6. easy 곡 리스트가 별도 섹션에 표시되어야 한다
7. 원클릭 신청 버튼이 동작해야 한다
8. 원클릭 신청 시 `참여자`, `세션`은 필수이고 `비고`는 선택 입력으로 받는다
9. 사용자는 본인이 작성한 신청곡뿐 아니라 다른 신청곡에도 추가 참여 멤버 정보를 붙일 수 있어야 한다

### 선택 기능

1. 신청 삭제 또는 수정
2. 희망 세션별 필터
3. 신청 마감 여부 토글
4. 관리자 전용 easy 곡 리스트 수정 기능

## 11. 비기능 요구사항

### 사용성

1. 모바일에서도 사용할 수 있어야 한다
2. 첫 방문자도 설명 없이 이해 가능한 구조를 지향한다
3. 신청 과정은 1분 이내 완료 가능해야 한다

### 유지보수성

1. 코드 구조가 단순해야 한다
2. 컴포넌트와 데이터 구조가 명확해야 한다
3. 추후 카카오톡 연동이 가능하도록 확장 여지를 남긴다

### 신뢰성

1. 입력값 검증이 필요하다
2. 링크 형식은 최소한으로 검증한다
3. 데이터 손실 위험이 적은 구조를 택한다

## 12. UI/UX 방향

### 디자인 원칙

1. 복잡한 관리 툴보다는 "동아리용 게시판 + 신청 폼" 느낌
2. 밝고 친근한 분위기
3. 모바일 우선 레이아웃

### 중요한 UX 포인트

1. 표만 덩그러니 두지 말고, 바로 아래에 신청 폼을 둔다
2. 사용자가 무엇을 해야 하는지 버튼과 문구로 분명히 안내한다
3. 곡을 못 고른 사람을 위한 다음 행동이 항상 보이게 한다

### 카피 초안 예시

1. `이번 잼 데이, 어떤 곡으로 함께할까요?`
2. `이미 정한 곡이 있다면 바로 신청하고, 없다면 easy 리스트를 참고해보세요.`
3. `그래도 고민된다면 원클릭으로 운영진 추천을 받을 수 있어요.`

## 13. 개발 단계 제안

### 1단계: 기획 확정

상태: 완료

1. 필요한 필드 이름 확정
   - `곡명`, `신청자`, `희망 세션`, `추가 참여 멤버`, `비고(유튜브 링크 등)`
2. 희망 세션 종류 확정
   - `보컬`, `키보드`, `기타`, `베이스`, `드럼`, `색소폰`, `기타 악기`
3. easy 곡 리스트 운영 방식 결정
   - 우선은 `(곡명) - (가수명)` 다음 줄에 `(링크)`를 두는 단순한 형태로 MVP 구현
   - 추후 genre, 난이도 등 태그를 추가하고 표현 형태를 개선
4. 원클릭 신청 시 어떤 정보까지 받을지 결정
   - `참여자`, `세션`, `비고`
   - MVP에선 더미 알림까지만 구현하고, 추후 카카오톡 API와 연동해 관리자에게 전달되게 확장

### 2단계: 화면 프로토타입

1. 상단 소개 섹션
2. 신청 현황 표
3. 신청 추가 폼

## 15. 확정된 구현 기준

1단계 기획 확정 과정에서 아래 항목은 다음과 같이 결정되었다.

1. `희망 세션`은 자유 입력이 아니라 고정 목록을 사용한다
2. `추가 참여 멤버`는 한 줄 문자열이 아니라 `이름 + 세션` 구조화 데이터로 저장한다
3. 신청 수정/삭제 권한은 초기 MVP에서 모두에게 열어 둔다
4. 원클릭 신청 시 `참여자 이름` 입력은 필수다
5. easy 곡 리스트는 운영진이 수동으로 관리한다
6. 신청 데이터 공개 범위는 별도 내부 제한 없이 운영한다

이 결정에 따른 구현 메모:

1. 수정/삭제 권한을 모두에게 열어 두는 대신, MVP에서는 복잡한 인증 시스템 없이 간단한 사용 흐름을 우선한다
2. 공개 범위를 제한하지 않더라도, 실제 배포 시에는 민감한 개인정보를 받지 않는 방향으로 입력 항목을 최소화한다
3. 추가 참여 멤버를 구조화 데이터로 저장하면 이후 수정, 삭제, 세션별 집계 기능 확장이 쉬워진다

## 16. 성공 기준

이 프로젝트는 아래 조건을 만족하면 1차 성공으로 본다.

1. 동아리원이 실제로 곡을 신청할 수 있다
2. 운영진이 표만 보고 신청 현황을 파악할 수 있다
3. 곡을 못 고른 사람도 easy 리스트나 원클릭 신청으로 다음 행동을 할 수 있다
4. 개발자인 나중의 나도 코드를 다시 봤을 때 이해할 수 있다
